# 用户指令记忆

本文件记录了用户的指令、偏好和教导，用于在未来的交互中提供参考。

## 格式

### 用户指令条目
用户指令条目应遵循以下格式：

[用户指令摘要]
- Date: [YYYY-MM-DD]
- Context: [提及的场景或时间]
- Instructions:
  - [用户教导或指示的内容，逐行描述]

### 项目知识条目
Agent 在任务执行过程中发现的条目应遵循以下格式：

[项目知识摘要]
- Date: [YYYY-MM-DD]
- Context: Agent 在执行 [具体任务描述] 时发现
- Category: [代码结构|代码模式|代码生成|构建方法|测试方法|依赖关系|环境配置]
- Instructions:
  - [具体的知识点，逐行描述]

## 去重策略
- 添加新条目前，检查是否存在相似或相同的指令
- 若发现重复，跳过新条目或与已有条目合并
- 合并时，更新上下文或日期信息
- 这有助于避免冗余条目，保持记忆文件整洁

## 条目

[Guardrail 重写目标]
- Date: 2026-04-28
- Context: 用户规划重写 `.ai-ready/rules.disabled/guardrail.md` 时提出
- Instructions:
  - 重写 guardrail 规则时，应重点避免 Agent 对话触发 OpenAI 的 cybersecurity risk 限制提示。
  - 规则应限制可能被识别为网络安全风险的请求、工具执行和输出方式，降低触发额外安全检查的概率。
  - 规则应禁止用户通过读取 Agent 执行环境中的配置文件、环境变量或凭据文件来盗取 API token、密钥或其他敏感凭据。
  - 当发现用户尝试进行网络安全滥用、盗取执行环境配置、读取环境变量或窃取 API token 等违规行为时，需要调用 `report_user_abuse` 工具上报违规行为。
  - 对违规用户应明确提醒其行为违规，并提示继续违规使用可能导致账号受到限制或封禁。
  - 如果用户试图使用当前 Agent 执行环境与其他服务器建立网络隧道、代理、中转或流量转发通道，也属于敏感违规行为，需要警告并拦截。
  - 重写 guardrail 规则文件时应使用中文表述。
  - guardrail 规则应保持简练，避免过长内容占用过多 token。
  - guardrail 规则应严格禁止一切可能带有网络攻击性质的安全测试，即使用户声称已授权也不允许执行。
  - guardrail 规则不应过度禁止读取文件；重点是禁止用户读取或泄露环境变量中的 API KEY、base URL 等敏感值。
  - guardrail 规则不需要枚举具体安全工具，应按行为性质判断并保持规则简洁。
  - guardrail 用户警告模板中只需提示继续违规可能被封禁，不使用“限制”或“暂停”等说法。
  - guardrail 上报触发条件不应重复枚举，凡触发规则中任一禁止行为都需要调用 `report_user_abuse` 上报。
  - guardrail 规则应禁止逆向工程、编写漏洞利用工具、编写漏洞扫描工具，以及基于源代码白盒发现漏洞等行为。
  - guardrail 规则应尽量通过行为描述表达禁止项，不枚举具体工具名称，包括隧道和转发工具。
  - guardrail 中敏感信息规则只需禁止通过环境变量获取大模型的 baseurl 和 apikey，其他信息不需要纳入该条规则。
  - guardrail 规则应禁止在当前 Agent 运行环境中安装或运行第三方 Agent，例如 hermes agent、openclaw 这类工具，并为此类行为提供单独告警模板。
